# Синхронизация на основе UUID

## Архитектура

Новая система синхронизации использует UUID для идентификации записей и флаги состояния для отслеживания изменений.

### Ключевые изменения

1. **UUID для всех записей** — каждая запись получает уникальный UUID при создании
2. **Флаги синхронизации** — `_sync_status` и `_modified` для отслеживания состояния
3. **Двусторонняя синхронизация** — отправка изменений и получение обновлений с сервера
4. **Нет очереди операций** — вместо очереди отправляются все изменённые записи

---

## Структура записей

### IndexedDB (фронтенд)

```javascript
{
  uuid: "a3f5b8c1-2d4e-4f6a-9b8c-1d2e3f4a5b6c",  // Уникальный ID
  id: 42,                                          // Локальный ID (если есть)
  name: "Обогреватель",
  _sync_status: "pending",  // pending | synced | failed
  _modified: true,          // true если есть локальные изменения
  _sync_error: null,        // Текст ошибки если failed
  synced_at: "2026-03-11T14:00:00Z",
  created_at: "2026-03-11T13:00:00Z"
}
```

### PostgreSQL (бэкенд)

```sql
CREATE TABLE heaters (
  id SERIAL PRIMARY KEY,
  uuid UUID DEFAULT gen_random_uuid(),
  name VARCHAR(100),
  synced_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP
);
```

---

## Статусы синхронизации

| Статус | Описание | Что происходит |
|--------|----------|----------------|
| `pending` | Ожидает синхронизации | Новая запись или есть локальные изменения |
| `synced` | Синхронизирована | Запись на сервере, локальных изменений нет |
| `failed` | Ошибка синхронизации | Произошла ошибка, требуется повторная попытка |

---

## Визуальные индикаторы

В интерфейсе используются индикаторы статуса:

- ⏳ **Жёлтый** — запись ожидает синхронизации (pending)
- ❌ **Красный** — ошибка синхронизации (failed)
- (нет индикатора) — запись синхронизирована (synced)

## Процесс синхронизации

### 1. Фронтенд → Бэкенд

```javascript
// Получаем все изменённые записи
const payload = await Store.getSyncPayload();
// { heaters: [...], premises: [...], objects: [...], lastSyncTime: "..." }

// Отправляем на сервер
const response = await fetch('/api/sync', {
  method: 'POST',
  body: JSON.stringify(payload)
});

// Получаем обновления с сервера
const { data } = await response.json();
// data.heaters, data.premises, data.objects, ...

// Применяем к локальным данным
await Store.applyServerResponse(data);
```

### 2. Бэкенд → Фронтенд

```javascript
// Сервер получает lastSyncTime
const lastSyncTime = payload.lastSyncTime;

// Возвращает все записи, изменённые после этого времени
const updates = await query(`
  SELECT * FROM heaters 
  WHERE created_at > $1 OR synced_at > $1
`, [lastSyncTime]);
```

## API

### Store.js

```javascript
// Создание записи (с UUID и флагами)
await Store.create('heaters', { name: "...", object_uuid: "..." });

// Обновление записи
await Store.update('heaters', uuid, { name: "Новое имя" });

// Удаление записи
await Store.delete('heaters', uuid);

// Получить pending записи
const pending = await Store.getPending('heaters');

// Пометить как синхронизированную
await Store.markSynced('heaters', uuid, serverData);

// Пометить как ошибку
await Store.markSyncError('heaters', uuid, "Ошибка сети");

// Получить payload для отправки
const payload = await Store.getSyncPayload();

// Применить ответ сервера
await Store.applyServerResponse(serverData);
```

### SyncManager.js

```javascript
// Основная синхронизация
await SyncManager.sync();

// Принудительная синхронизация
await SyncManager.forceSync();

// Авто-синхронизация при появлении интернета
SyncManager.init();
```

## Примеры использования

### Создание объекта оффлайн

```javascript
// 1. Создаём объект (оффлайн)
await Store.create('objects', { name: "Тестовый объект" });
// _sync_status: 'pending', _modified: true

// 2. Проверяем статус
const pending = await Store.getPending('objects');
// [{ uuid: "...", name: "Тестовый объект", _sync_status: "pending" }]

// 3. Появился интернет — синхронизируем
await SyncManager.sync();
// Отправляет на сервер, получает server ID
// _sync_status: 'synced', _modified: false
```

### Обновление записи

```javascript
// 1. Обновляем обогреватель
await Store.update('heaters', uuid, { status: 'repair' });
// _sync_status: 'pending', _modified: true

// 2. Синхронизируем
await SyncManager.sync();
// Сервер обновляет запись
// _sync_status: 'synced', _modified: false
```

### Обработка ошибок

```javascript
// Если синхронизация не удалась
try {
  await SyncManager.sync();
} catch (err) {
  // Записи помечаются как failed
  // _sync_status: 'failed', _sync_error: "Failed to fetch"
  
  // Повторная попытка
  await SyncManager.sync();
}
```

Выполните миграцию для добавления UUID:


- `uuid UUID` — уникальный идентификатор

## Отличия от старой системы

| Старая система | Новая система |
|----------------|---------------|
| Локальные ID (`local_123`) | UUID v4 |
| Очередь операций | Флаги `_sync_status` и `_modified` |
| Сложная обработка зависимостей | Разрешение UUID на сервере |
| Только фронтенд → бэкенд | Двусторонняя синхронизация |
| Нет статуса у записей | Статус у каждой записи |

## Решение проблем

### Запись не синхронизируется

1. Проверьте статус: `record._sync_status`
2. Если `failed` — проверьте `_sync_error`
3. Попробуйте ещё раз: `SyncManager.sync()`

### "Failed to fetch"

- Backend недоступен
- Проверьте подключение к серверу
- CORS настройки в `server.js`

### Дубликаты записей

- UUID должен быть уникальным
- Проверьте индексы в БД: `idx_heaters_uuid`

## Исправленные ошибки (версия от 2026-03-11)

### 1. Исправлена обработка UUID полей на бэкенде

**Проблема:** Бэкенд пытался вставить `_uuid` поля (например, `premise_uuid`, `object_uuid`) напрямую в БД, но этих полей нет в схеме PostgreSQL.

**Решение:** В `sync-endpoint.js`:
- Удалены `_uuid` поля из списка полей для вставки
- Добавлена функция `resolveReferences()` для разрешения UUID ссылок перед вставкой
- UUID поля теперь используются только для поиска соответствующих ID

**Файлы:** `backend/sync-endpoint.js`

### 2. Добавлена поддержка soft delete (`deleted_at`)

**Проблема:** Фронтенд использовал soft delete, но бэкенд не синхронизировал поле `deleted_at`.

**Решение:**
- Добавлено поле `deleted_at` во все основные таблицы
- Обновлены функции `createRecord()` и `updateRecord()` для обработки `deleted_at`
- Обновлены запросы `getServerUpdates()` для фильтрации удалённых записей

**Файлы:** `backend/sync-endpoint.js`, `schema.sql`, `migration_soft_delete.sql`

### 3. Удалено дублирование endpoint'ов

**Проблема:** Endpoint'ы `/stickers` и `/events` были определены дважды в `api.js`.

**Решение:** Удалены дублирующиеся определения после подключения `sync-endpoint.js`.

**Файлы:** `backend/api.js`

### 4. Исправлена генерация UUID на фронтенде

**Проблема:** Функция `generateUUIDSync()` генерировала UUID в неправильном формате (не соответствовал стандарту UUID v4).

**Решение:** Исправлен формат генерации UUID — теперь позиция версии (4) и варианта (8-9-a-b) устанавливаются корректно.

**Файлы:** `public/store.js`

### 5. Обновлена схема БД

**Проблема:** `schema.sql` не содержала UUID полей и некоторых индексов.

**Решение:** Полностью переписан `schema.sql` с включением:
- UUID полей для всех таблиц
- `deleted_at` для soft delete
- `synced_at` для синхронизации
- Всех необходимых индексов

**Файлы:** `schema.sql`

### 6. Исправлена работа Store.update() и Store.delete() с ID записей

**Проблема:** Функции `Store.update()` и `Store.delete()` искали записи только по UUID, но записи с сервера могут иметь только `id` без `uuid`.

**Решение:**
- Добавлен поиск записей как по UUID, так и по ID
- Обновлены функции: `update()`, `delete()`, `markSynced()`, `markSyncError()`
- Теперь редактирование объектов и помещений работает корректно

**Файлы:** `public/store.js`

### 7. Обновлена версия Service Worker

**Проблема:** Service Worker кэшировал старую версию `store.js`.

**Решение:** Обновлена версия кэша с `electro-v1` на `electro-v2`.

**Файлы:** `public/sw.js`

### 8. Исправлена синхронизация UUID помещений и объектов для обогревателей

**Проблема:** При синхронизации на другом устройстве обогреватели не получали `premise_uuid` и `object_uuid`, поэтому названия вагонов отображались неправильно.

**Решение:**
- Бэкенд теперь возвращает `p.uuid as premise_uuid` и `o.uuid as object_uuid` в API обогревателей
- Обновлены endpoint'ы: `/api/heaters` и `/api/heaters/:id`
- Обновлён sync-endpoint для возврата UUID в `getServerUpdates()`
- Фронтенд правильно сохраняет `premise_uuid` и `object_uuid` при синхронизации

**Файлы:** `backend/api.js`, `backend/sync-endpoint.js`, `public/store.js`, `public/sw.js`

### 9. Устранено дублирование помещений после синхронизации

**Проблема:** При добавлении обогревателя с последующей синхронизацией создавались дубликаты помещений — одно локальное "(офлайн)" и одно серверное с заметкой.

**Решение:**
- `syncFromServer()` теперь ищет локальные записи как по UUID, так и по ID
- Добавлен поиск дубликатов помещений по имени и object_id/object_uuid
- При обнаружении дубликатов (одинаковое имя + объект):
  - Обогреватели переназначаются на серверное помещение с UUID
  - Локальный дубликат без UUID удаляется
- `refreshPremises()` и `refreshObjects()` выполняют дедупликацию — предпочитают записи с UUID
- Локальные изменения сохраняются при слиянии
- Добавлена функция `cleanDuplicatePremises()` для ручной очистки дубликатов

**Файлы:** `public/store.js`, `public/app.js`, `public/sw.js`

### 10. Исправлено отображение индикаторов синхронизации и Инв. №

**Проблема:**
- После синхронизации исчезал "Инв. №" (sticker_number), но оставались песочные часы ⏳
- Должно быть наоборот: Инв. № остаётся, песочные часы исчезают

**Решение:**
- `renderHeaterItem()` теперь показывает индикатор синхронизации только по `_sync_status`, игнорируя `_modified`
- `mapServerRecord()` сохраняет `sticker_number` с сервера
- При слиянии дубликатов обогреватели получают `_sync_status: 'synced'` и `_modified: false`
- `refreshHeaters()` получает `sticker_number` из таблицы `stickers` через мапу
- Локальный `sticker_number` сохраняется при синхронизации
- Наклейки создаются со статусом `pending` для синхронизации с сервером

**Файлы:** `public/app.js`, `public/store.js`, `public/sw.js`

### 11. Добавлена кнопка удаления обогревателя для админа

**Изменения:**
- В `renderHeaterItem()` добавлена кнопка 🗑️ для пользователей с ролью `admin`
- В `renderListView()` (таблица) также добавлена кнопка удаления для админа
- Кнопка вызывает существующую функцию `deleteHeater()`
- Кнопка не перекрывает клик по элементу списка (`event.stopPropagation()`)
- Выполняет soft delete — запись помечается `deleted_at` и может быть восстановлена

**Файлы:** `public/app.js`, `public/sw.js`

### 12. Добавлена возможность редактирования обогревателей в офлайн

**Проблема:** При создании обогревателя офлайн его нельзя было отредактировать до синхронизации.

**Решение:**
- `Store.create()` теперь возвращает `id` присвоенный Dexie при добавлении
- `showEditHeaterModal()` ищет обогреватель:
  1. В `window.heaters` по `id`
  2. В IndexedDB по `id`
  3. В IndexedDB по `uuid` или `id` (полный поиск)
  4. На сервере через API (если онлайн)
- `Store.update()`, `Store.delete()`, `markSynced()`, `markSyncError()` используют `toArray()` для поиска, чтобы избежать ошибок Dexie с невалидным ключом
- После редактирования обновляется `selectedHeater` и открытая карточка
- Добавлено логирование для отладки

**Файлы:** `public/store.js`, `public/app.js`, `public/sw.js`

### 13. Обновление открытой карточки после редактирования

**Проблема:** После редактирования обогревателя данные сохранялись, но в открытой карточке не обновлялись.

**Решение:**
- `handleEditHeater()` обновляет `selectedHeater` после сохранения
- При следующем `render()` отображаются актуальные данные

**Файлы:** `public/app.js`, `public/sw.js`

### 14. Добавлено ведение журнала событий обогревателей

**Проблема:** История обогревателя была пустой, события не создавались.

**Решение:**
- При создании обогревателя создаётся событие с комментарием "Обогреватель создан"
- При редактировании создаётся событие с перечислением изменений (название, статус, помещение)
- При удалении создаётся событие с комментарием "Обогреватель удалён"
- События сохраняются в IndexedDB и синхронизируются с сервером
- `loadHeaterEvents()` ищет события по `heater_id` или `heater_uuid`

**Файлы:** `public/app.js`, `public/sw.js`

### 15. Добавлена дата изменения в таблицу обогревателей и быстрая смена статуса

**Проблема:**

- Не было быстрого способа изменить статус обогревателя

**Решение:**

- В карточке обогревателя добавлены кнопки для быстрой смены статуса
- `changeHeaterStatus()` меняет статус и создаёт событие в истории
- Событие содержит старый и новый статус с комментарием

**Файлы:** `public/store.js`, `public/app.js`, `public/sw.js`

### 16. Исправлено закрытие множественных модальных окон

**Проблема:** При открытых двух модальных окнах (карточка обогревателя и история) кнопка закрытия закрывала сначала карточку, а не историю.

### 17. Добавлена сортировка вагонов (помещений)

**Проблема:** Не было возможности сортировать вагоны в режиме "Вагоны".

**Решение:**
